# Unifying Thread + Affinity Registry

This skill combines cross-domain unification with a dynamic **Affinity Registry** — a living map of relationships, synergies, and high-leverage connections between skills, projects, concepts, deployments, and governance layers in your sovereign ecosystem.

It is **callable** (MCP/skill API, webhook triggers, A2A event patterns) and **auto-sensing** (detects when unification or affinity mapping is needed from conversation context, GitHub activity, PBC events, skill deployments, etc.).

## Activation Triggers (Auto-Sensing + Callable)
**Auto-sensing contexts** (the skill activates proactively when these patterns appear):
- Multiple skills/projects/concepts mentioned without clear connection (e.g., skill80-20 + PQC + ZK graph proofs + KVDF licensing + PBC).
- GitHub deployments, skill registrations, or unifying-thread invocations.
- PBC/EULER activations, partner role changes, or content deployments.
- Fragmented workflows across recommendation signals, ZK proofs, post-quantum attestation, skill orchestration, and physics-informed goals.
- Deployment health analysis or collaborator goal mapping.

**Callable patterns** (explicit invocation):
- "unifying-thread", "affinity registry", "map affinities between [X] and [Y]"
- a2a event unifying-thread-affinity-registry
- webhook unifying-thread
- mcp dispatch unifying-thread-affinity-registry
- github deployment strategy unification / github prep unifying thread

## Core Workflow (Enhanced with Affinity Registry)
1. **Auto-detect mountain** — Sense fragmentation or multi-domain context automatically.
2. **Establish goal frame** — Hard invariant (gravity theory) + soft objectives (13 revenue streams, sovereign infrastructure, PBC governance, hybridization).
3. **Build / Query Affinity Registry** — Map connections across:
   - Shared mechanisms (e.g., ZK proofs + PQC signatures + VDF in KVDF licenses)
   - Resources (skill80-20 distillation feeding graph intelligence)
   - Audiences (PBC partners, coalition members)
   - Failure modes / constraints (entropy overload → 80/20 pruning)
   - Inputs/outputs (recommendation signals → ZK graph proofs → soulbound KVDF licenses)
4. **Stress-test affinities** — Validate with evidence from deployments, evt- logs, and prior unifying-thread records.
5. **Output structured affinity map + through-line** — JSONL handoff + plain synthesis.
6. **Recommend next move** — Prioritized action that advances the goal hierarchy (e.g., register new skill, deploy to GitHub/Chain/Notion, create Retool dashboard).

## Affinity Registry Structure (Living Map)
The registry tracks affinities as structured records (stored in memory or evt- artifacts):
- Skill-to-Skill (e.g., skill80-20 ↔ unifying-thread ↔ PQC/ZK layer)
- Skill-to-Project (KVDF licensing ↔ Genesis Conductor MCP)
- Concept-to-Deployment (PQC attestation ↔ GitHub deploys + Chain on-chain)
- Governance affinities (PBC roles ↔ soulbound licenses)

Auto-updates on skill deployments, GitHub pushes, and PBC events.
